## TL;DR

The `sync.RWMutex` on `pipeline` and `noopPipeline` in
`pkg/logql/log/pipeline.go` is vestigial defensive code. It does not
provide actual thread safety (the double-check locking is incomplete,
and it doesn't cover `baseBuilder.Hash`/`baseBuilder.ForLabels`
which mutate shared state), and it contradicts the long-standing
documented contract at `pkg/ingester/tailer.go:190` that pipelines
are not thread safe. The sibling types in the same file
(`lineSampleExtractor`, `labelSampleExtractor`) have the exact same
per-stream cache pattern but no mutex, and never have.

I'd like to propose removing the mutex and documenting the contract
consistently. Happy to send a PR — wanted to flag the design
question first since the mutex was originally added in #9949.

All file references in this issue are permalinks pinned to
[`8553815420`](https://github.com/grafana/loki/tree/8553815420) so
they survive line-number drift.

## The structural twin argument

The strongest reason this is worth doing: `pkg/logql/log` already
has **four** types with structurally identical per-stream caches.
Two of them have a mutex, two of them don't.

**With a mutex** — `pkg/logql/log/pipeline.go`:
- [`noopPipeline.ForStream`](https://github.com/grafana/loki/blob/8553815420/pkg/logql/log/pipeline.go#L60-L77)
- [`pipeline.ForStream`](https://github.com/grafana/loki/blob/8553815420/pkg/logql/log/pipeline.go#L187-L204)

**Without a mutex** — `pkg/logql/log/metrics_extraction.go`:
- [`lineSampleExtractor.ForStream`](https://github.com/grafana/loki/blob/8553815420/pkg/logql/log/metrics_extraction.go#L77-L90)
- [`labelSampleExtractor.ForStream`](https://github.com/grafana/loki/blob/8553815420/pkg/logql/log/metrics_extraction.go#L188-L200)

All four follow the same pattern:

```go
func (x *T) ForStream(labels labels.Labels) U {
    hash := x.baseBuilder.Hash(labels)
    if cached, ok := x.cache[hash]; ok {
        return cached
    }
    res := newThing(..., x.baseBuilder.ForLabels(labels, hash))
    x.cache[hash] = res
    return res
}
```

They're called from the same code paths in the ingester:
- [`instance.go:525`](https://github.com/grafana/loki/blob/8553815420/pkg/ingester/instance.go#L525)
  — log query path uses `pipeline.ForStream(stream.labels)`
- [`instance.go:595`](https://github.com/grafana/loki/blob/8553815420/pkg/ingester/instance.go#L595)
  — metric query path uses `extractor.ForStream(stream.labels)`

Both iterate via `forMatchingStreams` which is a plain sequential
`for` loop. They have identical concurrency characteristics. If one
should have a mutex, the other should too. If neither should,
neither should.

The sample extractors have **never** had a mutex. `git log -S` on
both files since 2020 shows zero mutex-related commits. They follow
the "not thread safe, serialize at the caller" contract natively.

The `pipeline` mutex is the anomaly in this package, not the
standard.

## History: how the anomaly got there

The contract was established in
[#2917 (Nov 2020, @cyriltovena)](https://github.com/grafana/loki/pull/2917)
— "Fixes race condition in tailer since logql v2." The PR fixed a
real `go test -race` data race in `tailer.processStream`. The fix
was scoped entirely to `pkg/ingester/tailer.go` (+46/−7 lines across
`tailer.go` and `tailer_test.go`). It added:

- `t.pipelineMtx` — an external mutex in the tailer
- The comment `"pipeline are not thread safe and tailer can
  process multiple stream at once"` at
  [`tailer.go:190`](https://github.com/grafana/loki/blob/8553815420/pkg/ingester/tailer.go#L190)
- [`Test_TailerSendRace`](https://github.com/grafana/loki/blob/8553815420/pkg/ingester/tailer_test.go#L204-L227)
  — the concurrent regression test that still exists today

**Zero changes to `pkg/logql/log/pipeline.go`.** The design decision
was to serialize at the caller, not at the pipeline.

[#9949 (Jul 2023)](https://github.com/grafana/loki/pull/9949) —
"reset pipeline caches when tailing rather than making a new
pipeline" — added `Reset()` to address tailer memory growth.
#8890's short-circuit parsing optimization (Apr 2023) had made
pipeline construction expensive, and recreating pipelines per push
was OOM-ing the ingester when tailing high-volume streams. `Reset()`
let the tailer reuse one pipeline instance across streams.

As a side effect, #9949 also added `sync.RWMutex` to `pipeline` and
`noopPipeline`. The PR description never mentions thread safety.
The review discussion is two approvals with "looks good!" and no
comments on the mutex. No concurrent tests were added. The
`tailer.go:190` contract was not updated.

`git log -S 'sync.RWMutex' -- pkg/logql/log/pipeline.go` since then
shows only #9949 itself (plus its release-branch backport). The
mutex has not been touched in 2+ years.

## The mutex does not actually provide thread safety

If someone wanted to rely on it, it wouldn't work. Three bug
surfaces lie outside the protected section:

1. **Double-check locking without re-check.** `ForStream` takes the
   RLock, checks the cache, releases, takes the Lock without
   re-checking. Two concurrent callers can both miss the cache and
   race to overwrite each other.

2. **`baseBuilder.ForLabels` is called outside the write lock.**
   [`BaseLabelsBuilder.ForLabels`](https://github.com/grafana/loki/blob/8553815420/pkg/logql/log/labels.go#L175-L190)
   writes `b.resultCache[hash] = labelResult` without
   synchronization. Concurrent callers would hit
   `fatal error: concurrent map writes`.

3. **`baseBuilder.Hash` is called before any lock.** Under the
   default
   [`stringlabels`](https://github.com/grafana/loki/blob/8553815420/pkg/logql/log/labels_stringlabels.go#L16)
   build, `Hash` is stateless — it delegates to
   `labels.StableHash`. Under
   [`-tags slicelabels`](https://github.com/grafana/loki/blob/8553815420/pkg/logql/log/labels_slicelabels.go#L20),
   `hasher.buf` is a shared mutating scratch buffer, which is the
   exact race category #2917 was documenting. The Makefile ships
   with `-tags netgo` (no `slicelabels`), so production binaries
   use `stringlabels` and this particular race is latent in a
   non-default build.

A correct internal-mutex fix would need to cover all three. The
current mutex covers none of them completely.

## Why no production caller hits any of these races

I traced every production call site I could find. I could not
identify a caller that invokes `ForStream` concurrently on a shared
pipeline instance:

- **Log query path**
  ([`instance.go:525`](https://github.com/grafana/loki/blob/8553815420/pkg/ingester/instance.go#L525))
  — each query gets a fresh `*pipeline` via `expr.Pipeline()`
  ([`ast.go:214`](https://github.com/grafana/loki/blob/8553815420/pkg/logql/syntax/ast.go#L214)
  always returns `log.NewPipeline(stages), nil`).
  `forMatchingStreams` iterates sequentially.
- **Metric query path**
  ([`instance.go:595`](https://github.com/grafana/loki/blob/8553815420/pkg/ingester/instance.go#L595))
  — same structure with a fresh extractor per query.
- **Tailer path**
  ([`tailer.go:196`](https://github.com/grafana/loki/blob/8553815420/pkg/ingester/tailer.go#L196))
  — reuses one pipeline across streams but wraps the entire
  operation in `t.pipelineMtx.Lock()` at line 191.
- **Storage path**
  ([`store.go:500`](https://github.com/grafana/loki/blob/8553815420/pkg/storage/store.go#L500)
  →
  [`batch.go:419`](https://github.com/grafana/loki/blob/8553815420/pkg/storage/batch.go#L419))
  — `logBatchIterator.buildIterators` is a plain sequential `for`
  loop. The one goroutine in
  [`batch.go:741`](https://github.com/grafana/loki/blob/8553815420/pkg/storage/batch.go#L741)
  is for parallel chunk fetching and doesn't touch `ForStream`.
- **Query engine paths** — sharded sub-queries
  ([`downstream.go:529`](https://github.com/grafana/loki/blob/8553815420/pkg/logql/downstream.go#L529))
  are separate RPCs; BinOp parallelism
  ([`evaluator.go:832`](https://github.com/grafana/loki/blob/8553815420/pkg/logql/evaluator.go#L832))
  uses `errgroup` but each leg processes a different
  sub-expression, producing different pipeline instances.
- **Low-concurrency callers** — `pkg/logcli`, `pkg/bloombuild`,
  `pkg/chunkenc`, `pkg/compactor/deletion` all use fresh pipelines
  or single-threaded loops.

If there's a caller I missed — especially one where a pipeline
instance is genuinely shared across goroutines — I'd want to hear
about it, because that would change the right fix.

## Proposal

Remove the internal `sync.RWMutex` from `pipeline` and
`noopPipeline`. Add doc comments to both types documenting them as
not safe for concurrent use, matching the existing `tailer.go:190`
contract. Net diff: ~25 lines removed, ~10 lines of doc comments
added. No behavior change for any current caller.

### What this is NOT

**Not making `BaseLabelsBuilder` properly thread-safe.** That would
be the right fix *if* someone had a concrete need to call
`ForStream` concurrently on a shared pipeline, but no caller does
today, and doing it properly would touch `labels.go`,
`labels_slicelabels.go`, and affect every caller of
`BaseLabelsBuilder` — including the hot paths in the ingester and
querier that don't need concurrent access. If a future contributor
adds a parallel stream-processing use case, the right call at that
point is either (a) serialize at the new caller using the existing
tailer pattern, or (b) introduce a thread-safe wrapper type for
the specific use case. That decision belongs to someone with a
concrete use case in hand.

**Not adding new tests.** The existing `pipeline_test.go` covers
the sequential happy path extensively and passes under
`go test -race` against the current tree (verified in both default
and `-tags slicelabels` builds). A concurrent test would be *wrong*
given the "not thread safe" doc comment we're adding.
`Test_TailerSendRace` continues to be the canonical regression
guarantee for the scenario that actually matters.

## Why post this as an issue instead of just a PR

Two reasons:

1. The change removes code originally added in #9949. Even though
   I've done my homework on why the mutex is vestigial, there may
   be context I'm missing — an incident that motivated the
   defensive addition, a planned feature that was going to need
   the thread safety, something I can't see from the code. I'd
   rather surface that before writing a PR.

2. The framing matters. If we agree this is a cleanup rather than
   a functional fix, the PR is a ~25-line deletion and should
   land quickly. If there's a reason to keep the mutex (or fix
   it properly), I'd rather find out before investing in the
   wrong PR.

## Alternative if removal isn't preferred

If keeping the mutex is preferred for defense-in-depth reasons
(e.g., "we might want thread-safe pipelines later and don't want
to re-add it"), I'm happy to send a smaller PR that just (a) adds
the doc comments matching `tailer.go:190` and (b) fixes the
double-check locking so the mutex is at least internally correct.
That still improves the state of things without the architectural
question, and leaves the thread-safety decision for later.
